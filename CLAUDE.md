## このファイルの使い方

このCLAUDE.mdは、B-Hallというシステムを構築するうえでAIが最初に読む設計思想の根幹だ。
コードを書く前に必ずここを読め。UIを作る前も、APIを設計する前も、ここに戻れ。
「なんとなく作る」な。B-Hallは思想があるシステムだ。

---

## システム定義

B-Hallは営業活動以外の企業バックオフィス全体を一つの基盤に統合するクラウド型WebOSだ。
人事、労務、総務、経理、財務、法務、文書管理、申請、承認、稟議、報告、改善活動、
経営管理、組織管理、投資判断補助、行政手続き管理、テンプレート・ナレッジ・フロー管理。
これら全部をバラバラのツールとして並べるのではなく、
一本の企業運営フローとして繋げることがB-Hallの存在意義だ。

---

## 絶対に忘れるな：B-Hallの核心

バックオフィスの全業務が「一つの企業運営フロー」として繋がっていること。
これだけだ。
機能が多いことは価値じゃない。繋がっていることが価値だ。

### 繋がりの具体例（実装時も常に意識せよ）

- 入社 → 労務手続き → 総務貸与物発行 → ITアカウント発行 → 雇用契約書回収 → 経営者確認
- 経費申請 → 承認 → 支払 → 会計反映 → キャッシュフロー更新
- 稟議 → 契約書添付 → 支払依頼 → 実行タスク生成 → 効果検証メモ
- 事故報告 → 改善提案 → 改善タスク化 → 再発防止確認

これをUIで切断するな。APIで孤立させるな。データモデルで分断するな。

---

## システムの目的（一言で）

「営業活動さえ回せば事業成長に集中できる状態」を企業に作ること。
そのためにバックオフィス全体を可視化・標準化・自動化・管理可能にする。

---

## 思想：5層モデル

B-Hallはバックオフィスを以下の5層で捉える。
コードを書く時もこのモデルで考えろ。

### Layer 1 — イベント（業務の発生起点）

設立、入社、退社、異動、契約締結、支払発生、請求発生、事故発生、
月次締め、決算、規程改定、投資判断、設備購入、監査対応

### Layer 2 — フロー（イベントに紐づく処理経路）

誰が起票するか / 誰が確認するか / 誰が承認するか /
何の書類が必要か / 何の法的手続きが必要か

### Layer 3 — 業務オブジェクト（フローの中を流れるもの）

タスク、申請、承認、稟議、報告、書類、会計取引、コメント、通知

### Layer 4 — 蓄積される経営情報

進捗、人員状況、手続き状況、コスト、キャッシュ、リスク、未処理件数、改善テーマ

### Layer 5 — 経営判断と改善活動

どこが詰まっているか / どこに投資すべきか / どのフローを改善すべきか / どの部門が弱いか

## アーキテクチャ：レイヤー構造

```
A. 経営レイヤー
   経営ダッシュボード / 投資判断支援 / 資金繰り可視化
   リスク一覧 / 部門別状態可視化 / 改善テーマ管理

B. 業務統制レイヤー
   タスク管理 / フロー管理 / 申請・承認 / 稟議 / 報告

C. 部門業務レイヤー
   人事 / 労務 / 総務 / 経理 / 財務 / 法務 / 文書
   IT・アカウント・資産管理

D. ナレッジ・標準化レイヤー
   テンプレート / 手順書 / マニュアル / 規程 / FAQ
   ガイド / チェックリスト

E. 基盤レイヤー
   ユーザー・組織・権限管理 / マルチテナント / 通知基盤
   ファイル基盤 / ログ基盤 / カスタム項目・フォーム基盤
   ワークフローエンジン
```

## ロール設計

| ロール | 主な権限範囲 |
|--------|------------|
| 経営者 | 原則全閲覧。ただし生データ羅列ではなく経営判断粒度に整理されたビューで |
| 役員 | 経営者に準じる |
| 管理者 | 全体統制・承認待ち・期限超過・部門別進捗 |
| 部門責任者 | 自部門の全操作 |
| 経理・人事・労務・総務・法務担当 | 自担当範囲の操作、関連情報の横断参照可 |
| 一般従業員 | 自分のタスク・申請・承認待ち |
| 閲覧専用 | 読み取りのみ |
| 監査・外部専門家 | 証跡・ログ・文書への限定アクセス |
| Backlly担当者 | 顧客横断での管理・ヘルスチェック |

**重要**：経営者ビューは「全件表示」ではない。判断に必要な粒度に整理された統合ビューにしろ。

## モジュール一覧と実装上の注意

### M01 ダッシュボード

ロール別に必要な情報だけ表示する。情報を詰め込むな。

**経営者が見るもの：**
現預金残高、月次収支、資金繰り予測、未承認稟議、高額支出予定、
未処理法定手続き、従業員増減、離職・休職動向、リスク報告件数、
改善提案件数、未完了重要タスク、契約更新期限、重要インシデント、部門別滞留

**管理者が見るもの：**
承認待ち一覧、期限超過一覧、手続き未完了一覧、部門別進捗、
日報・報告未提出、コメント未対応、月次締め状況

**担当者が見るもの：**
今日やること、自分のタスク、承認待ち・差戻し、未読コメント、
締切が近いもの、必須提出物

**Backllyが見るもの：**
顧客別ヘルススコア、詰まり企業一覧、要介入タスク、次に提案すべき改善テーマ

---

### M02 タスク管理（B-Hall全体の中核）

タスクはシステム全体のハブだ。申請、稟議、書類、会計、通知、すべてがタスクと繋がる。

**必須フィールド：**
タイトル、詳細、業務カテゴリ、サブカテゴリ、会社、部署、
対象者・対象従業員・対象取引先、起票者、担当者、確認者、承認者、
期限、優先度、ステータス、関連書類、関連申請、関連稟議、
関連取引、チェックリスト、発生起点、テンプレートID、繰り返し設定、通知設定

**ステータス：**
未着手 → 進行中 → 確認待ち → 承認待ち → 差戻し → 保留 → 完了 → 中止

**自動生成トリガー（イベント連動）：**
入社、退社、異動、契約更新、月初、月末、年末調整時期、決算時期、
設立直後、社保加入対象発生、助成金対応時期、高額支出発生、稟議承認完了後

**ビュー：**
一覧 / ボード / 期限ビュー / カード（カレンダービューは不要）

---

### M03 ワークフロー（申請・承認）

すべてのバックオフィス申請をデジタル化し、承認経路を標準化する。

**対応申請種別：**
経費、交通費、休暇、残業、出張、購買、契約確認、支払依頼、採用、
端末・アカウント発行、稟議起票、社内制度利用、自由申請

**必須機能：**
フォームビルダー、カスタム項目、添付、承認ルート設定、条件分岐承認、
再申請、差戻し、代理申請、代理承認、承認・申請履歴、通知、関連タスク自動生成

---

### M04 稟議

会社の意思決定を記録・可視化・統制する。

**管理項目：**
件名、背景、目的、内容、金額、投資回収見込み、リスク、効果、
関係部門、添付資料、承認ルート、決裁日、決裁結果、実行タスク紐付け

**稟議完了後は必ず連携せよ：**
契約書 → 支払依頼 → 実行タスク → 効果検証メモ

---

### M05 人事・労務

人に関するすべての手続き・状態・リスクを漏れなく管理する。

**従業員マスタ必須項目：**
基本情報、所属、役職、雇用形態、入退社日、契約情報、
社保・雇保状況、緊急連絡先、資格、健診情報、休職・復職状況、イベント履歴

**管理対象イベント：**
入社、退社、異動、昇格、雇用契約更新、社保加入・喪失、雇保加入・喪失、
労災手続き、年末調整、住民税、健康診断、ストレスチェック、就業規則改定、
有休付与・管理、休職・復職対応、36協定、各種法定帳票

**連動要件：**
イベント発生 → 自動タスク生成 → 必要書類チェック → 提出進捗管理 → 行政期限アラート

---

### M06 総務・庶務

日常の会社運営管理を標準化する。備品、貸与物、端末、アカウント、
オフィス設備、郵送物、来客、社内掲示、契約更新、行政提出文書、各種台帳。

---

### M07 法務・契約・文書

契約書、NDA、業委契約、雇用契約、就業規則、規程、届出書、
行政文書、証憑、稟議・申請添付文書を一元管理する。

**機能要件：**
アップロード、タグ、フォルダ、OCR前提構造、有効期限管理、
バージョン管理、閲覧権限、更新タスク自動生成、関連オブジェクト紐付け

---

### M08 経理・財務

経理実務と資金管理をバックオフィス全体と繋げる。孤立させるな。

```
取引管理    → 収入・支出・振替・勘定科目・補助科目・取引先・証憑紐付け
経費精算    → 申請・領収書・承認・精算状況・支払状況
請求管理    → 請求書作成・発行・送付・入金予定・消込・未収一覧
支払管理    → 依頼・承認・予定表・実行確認・未払一覧
月次管理    → 締めチェックリスト・試算表確認タスク・証憑回収・残高確認
CF管理      → 現預金残高・月別入出金予測・固定支出・臨時支出・資金繰りアラート
財務分析    → 月次収支・費目別・部門別・予算対比・未収未払・CF変動要因
```

**権限：** 金額閲覧制限あり。経営者は全閲覧。経理担当は操作可。部門別閲覧制御。

---

### M09 報告・改善

問題発見と改善活動を日常業務に組み込む。
日報・週報・月報・事故報告・インシデント報告・クレーム報告・
ヒヤリハット・改善提案・目安箱。

**フロー：** 投稿 → 対応者割当 → 改善タスク化 → 再発防止タスク化 → 経営者通知
匿名・実名切替機能を持つこと。

---

### M10 経営管理

経営者が会社全体の状態を一画面思想で把握し、
投資・改善・優先順位判断を行えるようにする。

売上・支出サマリー、資金繰り、未承認重要案件、高額稟議、
人員増減、離職・休職、部門別業務負荷、リスク件数、改善提案件数、
契約期限、法定手続き状況、重要未完了タスク、投資判断補助、
稟議とキャッシュの紐付け、投資後の実行タスク進捗、効果検証メモ。

---

### M11 ナレッジ・テンプレート

属人化を防ぎ、業務を標準化する。
手順書・業務マニュアル・ガイド・チェックリスト・コラム・
規程解説・テンプレート・FAQ。

タスク・申請・稟議から直接参照できること。バージョン管理必須。

---

### M12 通知・期限管理

**通知対象：**
タスク割当、期限接近・超過、承認依頼、差戻し、コメント、書類未提出、
契約更新期限、社保・雇保等行政期限、支払期限、入金遅延、
キャッシュ不足予兆、重要報告投稿

**通知方法：** アプリ内通知 + メール。将来の外部連携に備えた抽象化設計。

---

### M13 組織・権限・監査

組織管理：会社・部署・チーム・役職・ユーザー
権限種別：作成・編集・閲覧・承認・差戻し・経営情報閲覧・金額閲覧・
人事情報閲覧・設定変更

監査ログ：作成・編集・削除・承認・差戻し・アップロード・
権限変更・ログイン・エクスポート、すべて記録

## UI/UX 設計方針

### 目指す質感

Mac / iOSアプリのような静かな高級感。
ガラスコーティング感。半透明。ブラー。柔らかな境界線。ミニマル。
情報整理が上手い。触りたくなる業務システム。

### 構造ルール

- サイドバーを中核ナビにする
- 各モジュールはタブで整理
- カードUI中心。テーブルは最小限
- モーダルは情報過多にしない
- 空状態・エラー状態・承認待ち状態もデザインする
- スマホはカードとドロワー中心
- 1画面1目的
- 次にやることを最上部に出す

### 禁止事項（絶対に守れ）

- 安っぽいSaaSテンプレートの焼き直し
- 情報を詰め込みすぎたダッシュボード
- テーブルだらけの画面
- AI生成っぽい均一カード羅列
- どこかで見た管理画面

## 非機能要件

- クラウド型・マルチテナント・企業単位データ分離
- 高可用性・レスポンシブ
- 監査ログ完備
- 権限分離
- 検索性能（全文検索・フィルタ・ソート）
- 添付ファイル基盤
- 将来拡張性・モジュール追加性

## 開発時の判断基準（迷ったらここに戻れ）

**1. この機能は「繋がり」を強化するか？**
強化しないなら後回し。切断するなら作るな。

**2. 経営者が1画面で判断できるか？**
できないなら設計を見直せ。

**3. 担当者が自分の仕事だけに集中できるか？**
余計な情報が入ってきていないか確認しろ。

**4. イベントが発生したらタスクが自動で作られるか？**
手動入力で業務が始まる設計は失敗だ。

**5. このUIはMac標準アプリと比べて恥ずかしくないか？**
恥ずかしいなら作り直せ。

## このシステムが最終的に作りたい状態

経営者が朝ダッシュボードを開いたら、
会社の全体状況が一目でわかる。

担当者が出社したら、
今日やるべきことが全部リストに並んでいる。

バックオフィスで「あれ誰がやるんだっけ」「期限どうなってたっけ」
「この書類どこだっけ」が、永遠に起きない状態。

それがB-Hallだ。
